## 🔄 Nightly Evolution Loop

### **Training Schedule**
- **2:00 AM UTC:** Nightly LoRA distillation
- **Every 6 hours:** Pattern mining and analysis  
- **Every 12 hours:** Fallback prompt generation
- **Every 24 hours:** Job cleanup and maintenance

### **Training Flow**
```
Scheduler Triggers → Pattern Analysis → Generate LoRA Job 
     ↓
Queue in Redis → Trainer Worker Picks Up → QLoRA 4-bit Training
     ↓
Model Validation → Save to models_vol → Hot Reload API → Update Metrics

Budget Guard ──┐
Time Limits ───┼──→ Training Process
Error Handling ┘
```

### **Training Parameters**
- **Method:** QLoRA 4-bit quantization
- **Rank:** 16 (configurable)
- **Alpha:** 32 (configurable)
- **Time Limit:** 30 minutes per job
- **Budget Limit:** $0.10 per day (shared with API)
- **GPU Memory:** Dynamic allocation with monitoring

## 💰 Cost Protection System

### **Dual-Layer Budget Guards**

1. **Per-Request Cap: $0.02**
   - Blocks any single API call or trainer job exceeding 2¢
   - Protects against expensive model misconfiguration
   - Applied to both live traffic and training workloads

2. **Daily Budget Pool: $0.10**
   - Shared budget between API and training operations
   - Rolling 24-hour window tracking
   - Hard stop when limit reached

### **Cost Tracking Integration**
```python
# Automatic cost tracking on every operation
def update_cost(model: str, n_tokens: int, is_trainer: bool):
    cost_cents = PRICING_TABLE[model] * n_tokens
    
    # Guard 1: Per-request cap
    if cost_cents > 2.0:  # $0.02
        raise BudgetExceeded("Request exceeds per-call limit")
    
    # Guard 2: Daily budget check  
    if would_exceed_daily_budget(cost_cents):
        raise BudgetExceeded("Would exceed daily budget")
    
    # Record and proceed
    cost_ledger.debit(model, n_tokens, is_trainer)
```

### **CI Cost Protection**
- All commits tested against budget violations
- Evolution training tested in CI pipeline
- Cost regression tests run on every PR
- Release gate blocks deployment if budget exceeded

## 🚦 Canary Deployment System

### **Traffic Split Configuration**
- **Production:** 95% traffic to stable API
- **Canary:** 5% traffic to new version
- **Hot Traffic Switching:** Real-time weight adjustment
- **Health-Check Gating:** Canary must pass health checks

### **Deployment Commands**
```bash
# Deploy new canary version
make -f Makefile.spiral canary-deploy VERSION=v2.7.1

# Adjust traffic split (10% canary)
make -f Makefile.spiral set-canary-weight PERCENT=10

# Promote canary to main
make -f Makefile.spiral promote-canary

# Emergency rollback
make -f Makefile.spiral rollback
```

## 📊 Monitoring & Observability

### **Key Metrics**
- **Cost Tracking:** Real-time budget utilization
- **Training Success:** LoRA job completion rates
- **API Performance:** Latency, throughput, errors
- **GPU Utilization:** Memory, compute, fragmentation

### **Alerting Thresholds**
- **VRAM Warning:** > 8GB (alert), > 12GB (critical)
- **API Latency:** > 2000ms (warning), > 5000ms (critical)  
- **Budget Utilization:** > 80% (warning), > 95% (critical)
